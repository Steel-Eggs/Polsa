# Agent Bootstrap Checklist (Sputnik baseline)

Цель: быстро развернуть нового ассистента (как для проекта СПУТНИК), не теряя времени на поиски конфигов.

## 1. Первичная персонализация
1. **Файлы:** `IDENTITY.md`, `USER.md`.
2. **Что спросить у владельца:**
   - Имя, как обращаться, место/часовой пояс.
   - Семью, ключевые даты (дни рождения), адрес (если нужно для контекста).
   - Приоритеты (основные проекты, цели, табу).
   - Границы по внешним действиям (можно ли публиковать/писать от имени без разрешения?).
3. **Что сделать:**
   - Заполнить `IDENTITY.md` (имя ассистента, вайб, эмодзи, аватар).
   - Заполнить `USER.md` (контакты, семья, бизнес, приоритеты, правила).
   - Создать/обновить `memory/YYYY-MM-DD.md` с кратким журналом дня.
   - Удалить `BOOTSTRAP.md`, когда настройка завершена.

## 2. Доступ к терминалу / инструментам
1. **Цель:** включить `exec`, `process`, `read/write` и управление подагентами (tmux, sessions_spawn).
2. **Config:** файл `~/.openclaw/openclaw.json`.
3. **Шаги:**
   - Убедиться, что существует блок `tools`.
   - Включить обмен агентами:
     ```json5
     "tools": {
       "agentToAgent": { "enabled": true },
       ...
     }
     ```
   - Сменить профиль инструментов на «coding» или «full»:
     ```bash
     openclaw config set tools.profile "full"
     ```
     | Profile | Что включает |
     | ------- | ------------ |
     | minimal | только `session_status` |
     | messaging | `message` + сессии |
     | coding | `group:fs`, `group:runtime`, `group:sessions`, `group:memory`, `image` |
     | full | все инструменты |
   - Перезапустить: `openclaw gateway restart`.
4. **Проверка:**
   - `exec{"command":"pwd"}` возвращает `workspace`.
   - `sessions_list` / `sessions_spawn` доступны.

## 3. Голос / ElevenLabs TTS
1. **Найти и установить скилл:**
   ```bash
   clawhub search elevenlabs
   clawhub install openclaw-skill-elevenlabs-pro --force  # предварительно просмотреть код
   ```
2. **Подключить API-ключ:**
   - Сгенерировать ключ с правами `user_read`, `voices_read`, `speech_synthesis`.
   - Сохранить в `~/.openclaw/.env` (файл не коммитится):
     ```bash
     cat <<'EOF' > ~/.openclaw/.env
     ELEVENLABS_API_KEY=sk_...
     EOF
     ```
   - Перед запуском команд экспортировать переменную: `set -a && . ~/.openclaw/.env && set +a`.
   - Если API отвечает `missing_permissions`, попросить владельца пересоздать ключ с нужными правами.
3. **Подбор голоса:**
   - Прослушать предпросмотры (пример: River https://storage.googleapis.com/eleven-public-prod/premade/voices/SAz9YHcvj6GT2YYXdXww/e6c95f0b-2227-491a-b3d7-2249240decb7.mp3).
   - Команда для списка голосов и ссылок:
     ```bash
     python3 skills/openclaw-skill-elevenlabs-pro/scripts/elevenlabs.py --list-voices --json | less
     ```
   - Генерация тестового файла:
     ```bash
     python3 skills/openclaw-skill-elevenlabs-pro/scripts/elevenlabs.py "Текст" \
       --voice "River" --model eleven_multilingual_v2 \
       --stability 0.8 --similarity 0.7 --style 0.9 \
       --output media/tts/<имя>.mp3
     ```
   - **Текущий дефолт для Антона:** Voice River (ID `SAz9YHcvj6GT2YYXdXww`), `eleven_multilingual_v2`, stability 0.8, similarity 0.7, style 0.9, speaker boost включён.

## 5. (Резерв) новые настройки
- Каждую новую интеграцию описываем отдельным подразделом (что установить, какой конфиг, какие команды).
- Формат: *шаг → подзадачи → команды/файлы → проверка.*

_Последнее обновление: 2026-03-04 (Польза)._ 

## 4. Веб-поиск (Brave Search API)
1. **Получить API key:**
   - Зарегистрироваться на https://brave.com/search/api/ (потребуется карта, но ежемесячно дают $5 бесплатных кредитов).
   - Скопировать ключ формата `BSA0-...`.
2. **Прописать ключ в конфиге:**
   ```bash
   openclaw config set tools.web.search.apiKey "<BRAVE_API_KEY>"
   ```
   (Дополнительно можно держать копию в `~/.openclaw/.env` как `BRAVE_API_KEY=...`).
3. **Перезапустить gateway:** `openclaw gateway restart`.
4. **Проверка:** `web_search {"query":"пример"}` должно вернуть результаты Brave.

_Текущий ключ (март 2026): `BSA0-ydkBBsAiN-cvt0MbECGBKAN8A_`._

## 6. СПУТНИК: обязательные артефакты
- При запуске нового ассистента для СПУТНИКа сразу заводим отдельный AgentSkill (в каталоге skills/), чтобы базовая функциональность была доступна из коробки.
- Одновременно создаём/подключаем репозиторий на GitHub (версионирование, откаты, совместная работа). Минимум — init + push на GitHub; детали структуры и CI настраиваем по месту.
- Перед онбордингом проверяем, что skill и репозиторий прописаны в чек-листах и готовы к развертыванию.
# Регламент: Настройка сети и доступов для ассистентов (РФ-сегмент)

## 1. Проблема: Блокировка серверных IP (Datacenter IP)
* **Симптомы:** Российские хостинги (Sprinthost, Beget и др.) и сервисы Яндекса сбрасывают HTTP/HTTPS запросы (таймаут или Connection Reset), если они идут с серверных адресов зарубежных (или даже некоторых российских) дата-центров. VPN на сервере не решает проблему, так как IP всё равно остаётся серверным.
* **Решение:** Использование **Резидентных прокси (Residential Proxies)** с геопозицией "Россия".
* **Проверенный провайдер:** Успешное тестирование проведено на базе сервиса **Proxy-Seller** (proxy-seller.com / proxy-seller.io).
* **Нюансы настройки прокси (на примере Proxy-Seller):** 
  - Рекомендуется использовать авторизацию по белому списку IP (добавить IP сервера ассистента в кабинет провайдера).
  - При генерации прокси не ставить жесткую привязку к конкретному городу или региону, оставлять только страну — RU (Russia), чтобы избежать ошибок 407 из-за пустых пулов.
  - Обязательно использовать **Sticky-сессии** (липкие сессии), чтобы IP не менялся при каждом запросе и не вызывал подозрений у антифрод-систем.

## 2. Проблема: SmartCaptcha от Яндекса и защита от ботов
* **Симптомы:** Даже при использовании чистого резидентного IP, скриптовые браузеры (Playwright/Puppeteer в headless-режиме) детектируются Яндексом по техническим отпечаткам (fingerprint). В результате вместо страницы авторизации выдаётся непроходимая графическая капча "Я не робот".
* **Решение: Инъекция человеческих сессий (Cookie Injection).**
  - *Первичный вход:* Человек (администратор) заходит в аккаунт через обычный браузер (с включенным тем же резидентным прокси), проходит капчу, авторизуется и экспортирует Cookies в формате JSON.
  - *Работа ассистента:* Ассистент подгружает этот JSON в контекст браузера до перехода на сайт. Яндекс воспринимает это как продолжение доверенной сессии.
  - *Поддержание жизни (Важно!):* После завершения каждой задачи ассистент **обязан** собрать актуальные куки из текущего контекста браузера и перезаписать JSON-файл конкретного клиента. Это гарантирует, что сессия не протухнет и токены безопасности будут всегда свежими.

## 7. Тонкости общения и частые баги (Troubleshooting & Fixes)
При настройке нового ассистента необходимо сразу заложить в системный промпт (`IDENTITY.md` или `SOUL.md`) следующие правила для предотвращения типовых багов:

1. **Род и обращение:** ИИ (если выбран женский персонаж) должен строго следить за окончаниями глаголов, чтобы отвечать о себе в женском роде, а к владельцу (мужчине) обращаться в мужском.
2. **Форматирование ответов (Теги):** Если ассистент использует внутренние блоки рассуждений (например, `<think>...</think>` и `<final>...</final>`), необходимо строго соблюдать синтаксис. Любая опечатка (например, пропущенная угловая скобка) приведёт к вытеканию служебного кода в публичный чат (Telegram).
3. **Английский язык в ответах:** Строгий запрет на использование английского языка в публичных ответах (за исключением логов, кода и названий сервисов), если пользователь явно не попросил об этом. Весь вывод должен быть на русском языке.
4. **Проактивность при долгих задачах:** Ассистент не должен уходить "в закат" при запуске тяжелых скриптов (например, браузера). Сначала отправляется короткое сообщение «Взяла в работу», затем запускаются скрипты, затем выдаётся финальный ответ. Иначе у пользователя создаётся ощущение игнора, а предыдущее сообщение может не доставиться из-за прерывания сессии.
5. **Мониторинг лимитов (TPM/RPD):** При отладке кода, сборе скриншотов или парсинге тяжелых страниц контекст чата быстро раздувается. Нужно минимизировать передачу сырого HTML или логов в основной чат (сохранять их в файлы) и периодически сбрасывать контекст для экономии токенов.
6. **Ошибки тихих ответов (NO_REPLY):** Если ассистент использует инструменты, которые сами отправляют результат в чат (например, генерация голоса TTS), и пытается ответить системной командой `NO_REPLY` внутри тега `<final>`, платформа может не распознать команду и выдать пользователю системную ошибку "No response generated. Please try again." Нужно настраивать обработчик платформы так, чтобы он корректно "глотал" пустые текстовые ответы при наличии отправленных медиафайлов.
